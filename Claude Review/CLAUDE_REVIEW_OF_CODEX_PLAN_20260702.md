# Claude Review of Codex Plan: "Staged Reliability Program"
## July 2, 2026 - Third document; reviews the Codex implementation plan against CLAUDE_FINDINGS_AND_PLAN_20260702.md and CLAUDE_STANDARD_WORKFLOW_PLAN_20260702.md

## Verdict

The Codex plan is strong, correctly system-wide, and in several places better than the original proposals. Its architecture (shared TargetContext / RequirementElement / EvidenceRecord / EvidenceMatch / CoverageReport / BuildOutcome models feeding all five output types) is exactly the "one engine, five outputs" end state, and the fixture JDs (Foundant, Tyfone, Bonterra, IRS, one per lane) are used the right way: as regression anchors for shared logic, not as per-JD patches. Approve it with the additions below. Two of the gaps are S1 and should block the start of their respective phases.

## Codex's corrections to Claude's proposal: verified and accepted

All six corrections were checked against the repo and are right:

1. **Evidence catalog must index approved sources, not become a claim source.** Correct, and stricter than the original ontology design. Accepted as stated: config holds only synonym maps, capability tags, ownership limits, and never_claim patterns; facts stay in approved source files.
2. **Three title forms (official / display / matching).** Verified: "Associate Implementation Consultant - Strategic Advancement Focus" is the literal posting title in `jobs/job_description.txt`, so it is not removable noise. The three-form model is better than blanket suffix stripping. One rule to add: the display-title shortener must be conservative with meaningful specializations (an "ERP (Apparel)" or "Enterprise" qualifier changes what job it is; "Strategic Advancement Focus" does not). Make the shortener keep any segment whose words also appear in the JD's requirements sections, and drop only segments that appear nowhere outside the title line.
3. **Requirement projection instead of keyword extraction.** Accepted. The "adjust training is a real requirement concept but not a valid competency label" distinction is the correct resolution: represent it as a RequirementElement, never inject it as a competency.
4. **Tyfone public-agency question came from `application_questions.txt`.** Verified: it is line 7 of that file. Codex's diagnosis is better than the original "question bank tagging" fix, and DRAFT-plus-feedback instead of silent filtering is the right behavior for user-supplied input. See gap 7 for the missing lifecycle half of this fix.
5. **Dropping the 20% bullet-length rule and the blanket hyphenated-phrase ban.** Accepted. The targeted duplication/density/stacked-modifier checks preserve the intent without false positives on legitimate terms like "post-go-live" or "role-based."
6. **Coverage checks on both the content model and final rendered text.** Accepted; this is an improvement. Late formatting and scrub passes can alter text after the model check, so the rendered-text gate is the one that actually protects submissions.

Also verified: `python tasks.py validate` exists and maps to `scripts/smoke_test.py`, so the acceptance plan's entry point is real.

## Gaps to add before or during implementation

**S1 - Gap 1: Provenance of "Christian's confirmed facts" is asserted, not evidenced.** The plan writes three new fact sets into approved federal source JSON (East West stored procedures/views/functions/data models; Aderant hands-on backups, restores, recovery testing; Aptean Azure DevOps/TFS and Git/GitHub use) and its Assumptions section says the confirmations "authorize" the updates. This is the invented-claims risk zone and the one place the plan could silently violate the system's most important rule. Requirement: before the matcher phase starts, Christian explicitly re-confirms each of the three fact sets in writing (one message is enough), and the source JSON entries carry a `confirmed_on` date and the confirmation text. If any fact set was Codex's inference rather than Christian's statement, it comes out. The plan's own boundary language ("hands-on supports performing the activity, not administration leadership") is correct and should be encoded as the ownership limit on those records.

**S1 - Gap 2: The terminology injection mechanism is missing, so the federal coverage gate will fail with no way to pass.** The plan requires every Direct target-grade element to appear in dated work experience, but never says how the announcement's nouns get into the bullets. Without the injection stage, the IRS build simply hard-fails: the Home Depot bullets do not say "contact center," and nothing in the plan rewrites them. Add the companion document's Stage E explicitly: each EvidenceRecord carries allowed terminology and slot-based rewrite templates; the injector substitutes announcement-canonical terms only where the record lists them as allowed expressions, only inside the record's own role, preferring the most recent qualifying period. The acceptance plan needs a positive test to match: the regenerated IRS resume must contain "contact center," "service level," and "customer interaction data" inside the Home Depot block, sourced from the LivePerson/chat/SMS evidence record. Right now every IRS acceptance check is negative (no false claims, no $0); there is no check that supported coverage actually improved, which was the original point.

**S2 - Gap 3: Grade-equivalence framing (Stage G) is absent.** The one-year-at-GS-13-equivalent requirement is left to interpretation again. Add: detect the grade clause during federal parsing, and generate one framing sentence in the most recent role from scope facts on the evidence records (sites, users, independent authority). Template in config, facts from records, so it generalizes to every announcement and grade.

**S2 - Gap 4: Competency weaving (Stage H) stops at parsing.** The acceptance plan counts nine minimum and 14 assessed competencies parsed, but nothing weaves competency nouns into bullets or validates their presence. Add a warn-level registry rule: each announced competency name (or registered variant) should appear at least once in the document; report misses in the coverage matrix.

**S2 - Gap 5: The federal summary composer is not fixed by anything in the plan.** The stitched-clause grammar in `build_gs14_summary` ("delivering implementation, enterprise systems, and technology delivery and recent senior-technical-authority scope across... environments") is an S1 finding from the first document, and the Codex plan's prose framework covers cover letters, bullets, and spoken answers but never names the federal (or commercial) summary banks. Add: the summary composers migrate onto the artifact-specific assembler, the cluster `summary_phrase` fragments are rewritten as full slotted sentences, and the IRS regression gains a check that the regenerated summary passes the conjunction-overload and nested-list rules.

**S2 - Gap 6: DRAFT semantics must extend to the tracker and workflow runner.** The plan excludes DRAFT artifacts from downstream builders and from success reporting. Two consumers are unnamed: `track_applications.py` must not record a DRAFT build as an application-ready output, and `run_resume_workflow.py` must not count a DRAFT as a successful run for its automatic tracker updates. Name both.

**S3 - Gap 7: Application-questions lifecycle.** The stale-question defect has two halves. The plan covers detection (DRAFT plus feedback on mismatch). The prevention half: `tasks.py jd-archive` (and the workflow runner's JD-swap step) should archive or prompt about `jobs/application_questions.txt` alongside the JD, so a Foundant question file cannot silently survive into a Tyfone run. One-line change, removes the defect class at the source.

**S3 - Gap 8: Evidence intake loop.** The plan hardcodes today's three confirmations but drops the mechanism that generates tomorrow's: when the matcher classifies an element Unsupported or Adjacent, the CoverageReport should emit a plainly worded confirmation question for Christian. That loop is what makes the engine improve with every announcement instead of freezing at the IRS posting's gaps.

**S3 - Gap 9: Small items from the standard-workflow document not carried over.** Header segment semantic dedupe ("Enterprise SaaS | Enterprise Systems" must collapse; the three-title model fixes the suffix but not the redundancy). Bullet-ending punctuation normalizer. `.context` doc path corrections (`config/` vs the real `scripts/config/`), which matter because every future Claude review thread reads those docs first. Requirement-element-driven interview question prediction (an Adjacent element is a high-probability probing question and should get a prepared bridge answer); the plan routes elements to bridge decisions but not to question prediction.

## Acceptance plan additions

Beyond the positive IRS injection test (Gap 2), add: regenerated IRS and Foundant summaries pass the new structural prose rules; the IRS coverage matrix marks HA/DR clustering/replication and CI/CD ownership as honest non-claims while backup/recovery and version-control facts (if confirmed) appear under Aderant and Aptean respectively; a DRAFT resume in `output/` is ignored by a subsequent cover letter build (explicit test); `jobs/application_questions.txt` from a prior company triggers DRAFT with the suspect question named; header line for Foundant contains no two segments sharing a dominant word.

## Sequencing opinion

The order is right: defect shield first, ingestion second, matching third, prose/validation fourth, output integration fifth, model refactor last. The federal-first integration order is correct because the coverage gate is hard there and soft commercially, so it exercises the full engine earliest. One sequencing note: Gap 1 (confirmation provenance) blocks phase 3, and Gap 2 (injection) must land inside phase 3, not after it, or the federal gate cannot be turned on.

## Bottom line

Hitting all points: roughly 85% of the two Claude documents is fully covered, and every Codex deviation is an improvement, verified against the repo. The missing 15% clusters in the federal engine's second half (injection, grade framing, competency weaving, summary composition) - exactly the parts that turn a correct audit into a stronger resume, which was the original request. Add the nine gaps, adopt the acceptance additions, hold phase 3 until Christian re-confirms the three fact sets, and this plan is ready to implement.
